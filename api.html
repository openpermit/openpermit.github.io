API

Open API for standard interactions with a jurisdiction's permitting process.

Overview

Retrieve permit requirements, apply for permits, submit documents, retrieve issues, schedule inspections and a lot more.

Introduction

The Open Permit Specification presents permit requirements, plan submission standards, related forms and other relevant permit process information in a unified and standardized format. The API also covers operations for permit processes such as applying for a permit, submitting requirements (i.e. documents, construction plans, etc.), scheduling inspections and retrieving permit status information. These APIs will allow developers to build applications to streamline, automate and help visualize different aspects of the permitting process in jurisdictions.

Specification status

The Open Permit Specification is in current state of development, updates to the spec are posted here as we make progress to a stable API.

Open311

Where possible this specification will reference the Open311 spec to try to maintain consistency between the two APIs and minimize redundant effort.

Encoding

UTF-8 is required everywhere.

All text returned by the service, whether in JSON, XML, or any other text-based content-type, MUST be encoded as UTF-8. Appropriate HTTP headers must be set, and the root element must be preceded with the XML declaration including the encoding="UTF-8" attribute. All text received by the service from the client will be assumed to be UTF-8 and must be decoded accordingly. Examples of the HTTP content-type headers for each format are shown below under Format Support.

Format support

JSON

JSON is the default and only required format for the API and should be Content-Type: application/json.

XML

XML for the API should be Content-Type: text/xml. The XML output currently defined is schemaless and without a namespace declaration. For now, extending the output with additional namespaces should be done at your own risk. To guarantee that all clients can support the XML output, it is recommended that you adhere to the specification strictly and do not include foreign tags or namespaces. It is also assumed that the default namespace is not specified, but if it must be specified, the XMLNS URI is the URL of the specification: http://www.openpermit.org

Definitions

Optional Arguments

"Optional" means that the response should be the same whether a parameter is passed and is empty or is not passed at all. A null parameter should be treated as equivalent to a non-declared parameter in processing the request.

API Structure

The Open Permit API defines a large set of operations with the aim to support all parts of the permitting process in a jurisdiction. However, moving the entire permitting process to be electronic, all in one shot, could be quite a big task, not to mention too disruptive for many jurisdictions. We expect that many jurisdictions will not implement the entire specification in one pass, but they will gradually move part of their permitting workflows to be electronic. To this end, we have tried to divide the API into sections that can be implemented in isolation and will allow for gradual adoption of Open Permit. These sections have been aligned with specific services that can be offered by a jurisdiction individually, but could also all be part of a 100% electronic permitting process. In the future, we will define Open Permit implementation levels for an endpoint that will be determined by which sections of the API are supported by the endpoint. This will allow client to inquire about which specific services and information are supported by a jurisdiction online.

The areas supported by the API are the following:

  • Permit Section: a set of operations to retrieve information about specific permits and interact with the jurisdiction after the permit has been issued.
  • Requirements Section: a set of operations to retrieve all permitting requirements from a jurisdiction.
  • Plan Check Section: a set of operations to submit plan check requirements and interact with the jurisdiction in the plan check phase.

Permit Section

GET permits

Retrieves a list of permits.

Property Description
URL http://[API endpoint]/permits
Sample URL http://epermithub.com/permit?license=fl345654
HTTP Method GET
Requires API Key No

Arguments

Name Description Notes
address Address where permit is required. Optional Query Param.
license License number of the GC applying for permit. Optional Query Param.
related Permit number to query against to return its related permits. Optional Query Param.

Response

An array of {permit} objects with the following structure.

Property Description Notes
number Service endpoint assigned identifier for the permit. Required
type A {permit type}, see Permit Type structure below. Required.
state state of the permit: (application, review, inspection).
status The status of the permit given the state: (progress, done, issues).
address The address where the permit is needed.

Example

        name: 'Commercial Alteration'

    },

Retrieves the lists of all permit / project types for the jurisdiction.

Property Description
URL http://[API endpoint]/permits/types
Sample URL http://epermithub.com/permits/types
HTTP Method GET
Requires API Key No

Arguments

Name Description Notes

NO Arguments required

Response

An array of {permit type} objects with the following structure.

Property Description Notes
id Service endpoint assigned identifier for the permit type. Required
name Short description of the permit. Required.
desc Long description / help text for the permit type.
group A category to group this permit type within. This provides a way to group several permit types under one category.
