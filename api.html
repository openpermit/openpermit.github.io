APIs

Open APIs for standard interactions with jurisdiction's permitting processes.

Overview

Retrieve permit requirements, apply for permits, submit documents, retrieve plan check comments, schedule inspections and a lot more.

Introduction

The Open Permit Specification presents permit requirements, plan submission standards, related forms, code ordinances and other relevant permit process information in a unified and standardized format. The API also covers operations for permit processes such as applying for a permit, submitting requirements (i.e. documents, construction plans, etc.), scheduling inspections and retrieving permit status information. These APIs will allow developers to build applications to streamline, automate and help visualize different aspects of the permitting process in jurisdictions.

Specification status

The Open Permit Specification is in current state of development, updates to the spec are posted here as we make progress to a stable API.

Open311

Where possible this specification will reference the Open311 spec to try to maintain consistency between the two APIs and minimize redundant effort.

Encoding

UTF-8 is required everywhere.

All text returned by the service, whether in XML, JSON, or any other text-based content-type, MUST be encoded as UTF-8. Appropriate HTTP headers must be set, and the root element must be preceded with the XML declaration including the encoding="UTF-8" attribute. All text received by the service from the client will be assumed to be UTF-8 and must be decoded accordingly. Examples of the HTTP content-type headers for each format are shown below under Format Support.

Format support

XML

XML for the API should be Content-Type: text/xml. The XML output currently defined is schemaless and without a namespace declaration. For now, extending the output with additional namespaces should be done at your own risk. To guarantee that all clients can support the XML output, it is recommended that you adhere to the specification strictly and do not include foreign tags or namespaces. It is also assumed that the default namespace is not specified, but if it must be specified, the XMLNS URI is the URL of the specification: http://www.epermithub/op/index.html

JSON

JSON for the API should be Content-Type: application/json. JSON support is based on a programmatic mapping of the XML format using the Spark Convention. This means that if you have an XML file, you can easily generate the JSON version with XSL. This can be done using a server side XSLT.

PRDF

PRDF (short for Permit Requirements Description Format) defines a set of schemas to represent certain types of data returned or uploaded to the services. Currently these schemas will not be formally specified and will have no namespace, but their structure shall be enforced by the services. The specification will dictate which schema is supported by which service. Schemas are designed such that JSON support can be provided based on a programmatic mapping of the XML format using the Spark Convention.

Definitions

jurisdiction_id

As a means to allow this API to distinguish multiple jurisdictions within one API endpoint we've included a "jurisdiction_id" which will be the unique identifier for permitting issuing jurisdictions. It has been suggested that we use the jurisdiction's Census FIPS Code as defined by the US Census Permits Survey.

Optional Arguments

"Optional" means that the response should be the same whether a parameter is passed and is empty or is not passed at all. A null parameter should be treated as equivalent to a non-declared parameter in processing the request.

API Methos

GET checklists

Retrieves the lists of all permit / project types for a jurisdiction.

Property Description
URL http://[API endpoint]/checklists.[format]
Sample URL http://epermithub.org/checklists.json?jurisdiction_id=12095
Formats XML (JSON available if denoted by Service Discovery)
HTTP Method GET
Requires API Key No

Arguments

Name Description Notes
jurisdiction_id Id of jusrisdiction as defined by the service endpoint. This is only required if the endpoint serves multiple jurisdictions

Response

An array of <checklist> objects with the following structure.

Property Description Notes
id Service endpoint assigned identifier for the checklist. Required
name Short description of the checklist. Required.
desc Long description / help text for the checklist.
group A category to group this checklist within. This provides a way to group several checklists under one category. 
<checklists>
    ...
    <checklist>
      <id>54181792-ac3c-4694-9089-32aae77cf110</id>
      <name>Demolition</name>
      <desc>Minimum requirements for submittal of demolition permit.</desc>
    </checklist>
    <checklist>
      <id>742f92c1-04f3-42bd-b0e6-6821bf71a4d2</id>
      <name>Electrical</name>
      <desc>Minimum requirements for submittal of electrical permit.</desc>
    </checklist>
    <checklist>
      <id>c62c9d29-b085-45c7-b679-b91478603fe1</id>
      <name>Gas</name>
      <desc>Minimum requirements for submittal of a gas permit.</desc>
    </checklist>
    <checklist>
      <id>47c958fa-0f1e-4915-8749-f370c4b8bdcf</id>
      <name>New Commercial Structure</name>
      <code>Minimum requirements for submittal of new commercial structure building permit.</code>
    </checklist>
    ...
</checklists>

Possible errors

  • 404 - jurisdiction_id provided was not found (specify in error response)
  • 400 - jurisdiction_id was not provided (specify in error response)
  • 500 - General service error (Unknown error. The client will need to notify us)

GET checklist

Retrieves a permit / project type checklist containing the requirements for the specified permit / project type including documents to be submitted, fee schedule, plan review disciplines and subpermits if applicable.

Property Description
URL http://[API endpoint]/checklist.[format]
Sample URL http://epermithub.org/checklist.json?id=guid
Formats XML (JSON available if denoted by Service Discovery)
HTTP Method GET
Requires API Key No

Arguments

Name Description Notes
id Unique id for the permit checklist within this endpoint. Required

Response

The specified checklist in PRDF Checklist format.

Possible errors

  • 404 - checklist for id provided was not found (specify in error response)
  • 400 - id was not provided (specify in error response)
  • 500 - General service error (Unknown error. The client will need to notify us)

POST application

Submits a permit application supplying all the information needed to open a permit application for a particular type of permit. After this operation completes succesfully the permit record can be moved to a prescreen status where it is waiting for all the application requirements to be submitted or fulfilled.

Property Description
URL http://[API endpoint]/application.[format]
Sample URL http://epermithub.org/application.json
Formats XML (JSON available if denoted by Service Discovery)
HTTP Method POST
Requires API Key Yes

Arguments

Name Description Notes
fields List of fields that in PRDF Fields format. Required.

Response

Name Description Notes
permit_no The permit number associated with this application.

Possible errors

  • 404 - permit_no provided was not found (specify in error response)
  • 400 - permit_no was not provided (specify in error response)
  • 500 - General service error (Unknown error. The client will need to notify us)

GET runchecklist

Tallies all the items fulfilled for a particular permit application against it's checklist and returns the status of each checklist item as fulfilled or missing. When all requirements are fulfilled the permit record is ready to move to the next status and start the permitting process.
