PRDF

Permit Requirements Description Format, Open data format to consisely specify permit requirements.

Overview

Simple schemas for complex permit data representation.

Introduction

PRDF (short for Permit Requirements Description Format) defines a set of schemas to represent different types of permit related data. Complex relationships and permit process details can be represented using rules and formulas by employing a declarative approach. A format is also defined to represent a series of extenal fields or attributes that capture specific project information. In many of the higher level schemas, these fields are combined with rules and formulas to estimate things like specific permit requirements and applicable fees. Again all this information is stored in a declarative fashion and can be stored as data so that a generic software library can be used to process the applicable set of schemas and determine the outputs for specific cases by just receiving as input a few values unique to a given project.

Schema

Currently we have not defined any formal schema specs or namespaces. These may come in the future as the spec matures.

Open311

Where possible this specification will reference the Open311 spec to try to maintain consistency between the two APIs and minimize redundant effort.

Reusable formats

Introduction

PRDF defines a set of schemas, conventions and data formats that are used in building higher level formats for a variety of purposes.

Fields is a declarative schema that can be used specify miscelaneous required information and the type of that information. Fields can be used to gather information from users, to generate user interfaces and to filter the requiremens according to the conditions and formulas specified. Conditions is a declarative schema that allows for the definition of complex logic rules. They can be used to specify dependencies or alternative workflows of a given process or set of requirements. Formulas depict a simple syntax to represent calculations, much like the syntax of an Excel spreadsheet.

Fields

Fields are the most versatile constructs defined by PRDF. Think of fields as variables in a programming language or fillable fields in a Adobe PDF document. Fields are included in different higher level formats to represent external information that is needed to be able to accomplish a task, filter a set of requirements or simply evaluate a formula.

Open311

If you are familiar with the Open311 spec, fields serve a similar purpose as the attributes used to define services.

Inside the different schemas fields always come in their own <fields> section. 

   ...
  <fields>
    ...
  </fields>
  ...

Field

Fields is an array of <field> objects with the following structure.

Property Description Notes
name Name for the field. This is a string with no spaces, much like a variable name. Required property. Unique id.
label Short description or label for the field.
desc Long description with instructions or other help text
datatype Denotes type of data stored in the field.
  • boolean a true/false or yes/no value.
  • string a string of characters without line breaks.
  • text a string of characters that may contain line breaks.
  • numeric a numeric value.
  • list a set of predefined values (specified in this field) where only one value may be selected.
  • multilist a set of predefined values (specified in this field) where several values may be selected.
  • datetime the input generated must be able to transform into a valid ISO 8601 date.
 Possible values: boolean, string, text, numeric, list, multilist, dtatime.
items If the <datatype> property is list or multilist this property is an array of <item> strings that specify the possible values. Required if <datatype> is present. 
<fields>
    ...
    <field>
      <name>VALUATION</name>
      <datatype>numeric</datatype>
      <label>Project Valuation</label>
    </field>
    <field>
      <name>OWNERBUILDER</name>
      <datatype>boolean</datatype>
      <label>Owner Builder</label>
    </field>
    <field>
      <name>FNAME</name>
      <datatype>string</datatype>
      <label>First Name</label>
    </field>
    <field>
      <name>PROJDESC</name>
      <datatype>text</datatype>
      <label>Project Description</label>
    </field>
    <field>
      <name>STYPE</name>
      <datatype>list</datatype>
      <label>Structure Type</label>
      <items>
        <item>Residential Single family</item>
        <item>Residential Multi-family</item>
        <item>Commercial</item>
        <item>Mobile Home</item>
      </items>
    </field>
    ...
</fields>

Conditions

Conditions can be used to declaratively specify rules that dictate when some specific requirement or workflow step applies to a process. Conditions are added to specific items within different schemas to be able to filter them depending on the specific values provided by fields. An extensive set of boolean expressions is supported by conditions so that complex rules can be represented.

When conditions is supported on an item, you can add <conditions> to specify when the item applies.

   ...
  <conditions>
    ...
  </conditions>
  ...

Required items

When items that support <conditions> do not have one attached, it is assumed by convention that the item is required.

Condition

Conditions is an array of <condition> objects that are to be evaluated in order from first element to last. The <condition> object has the following structure.

Property Description Notes
field Name of field to evaluate. Required property.
value The value with which to compare the evaluated field value. Required property.
op Specifies which comparison operator to use to compare field and value
  • E equals operator.
  • NE not equals operator.
  • G greater operator.
  • GE greater or equal operator.
  • L lower operator.
  • LE lower or equal operator.
 Possible values: E, NE, G, GE, L, LE. Default value: E
logic Logic operator if specifying multiple conditions.
  • AND and operator.
  • OR or operator.
 Required if this is not the last condition. 
<conditions>
    <condition>
      <field>VALUATION</field>
      <op>G</op>
      <value>10000</value>
      <logic>AND</logic>
    </condition>
    <condition>
      <field>VALUATION</field>
      <op>LE</op>
      <value>25000</value>
    </condition>
</conditions>

<conditions> can be nested to express evaluation precedence. 

<conditions>
    <condition>
      <field>VALUATION</field>
      <op>G</op>
      <value>10000</value>
    </condition>
    <conditions>
      <condition>
        <field>STYPE</field>
        <value>Residential Single family</value>
        <logic>OR</logic>
      </condition>
      <condition>
        <field>STYPE</field>
        <value>Mobile Home</value>
      </condition>
    </conditions>
</conditions>

Formulas

Formulas, like conditions, can be attached to specific items that support them to have a value calculated on the fly. Formulas must follow the same syntax as Excel Formulas part of OOXML Standard (ECMA-376, ISO/IEC 29500-1:2008). If you prefer you can also follow OpenOffice formulas OpenDocument v1.2 OpenFormula (ODF v1.2 OASIS Standard). Only a subset of these specifications is needed, more work to be done exactly on what will be supported by Open Permit.

Formula tips

If you are not familiar with Excel formulas then a Google search should bring plenty of documentation on the subject.

Examples

2 + 3 * 4

((VALUATION * - 1000) / 1000) * 5

MAX(SQFEET * 10, 2500)

Checklist format

Structure

The checklist format can be used to publish specific permit requirement information including documents to be submitted, plan review disciplines that apply if any, applicable subpermits and fees. This format supports the definition of conditions and formulas that are tied to requirements so that the requirements can be filtered according to project details. Fields can also be defined that specify miscelaneous required information needed to filter the requiremens according to the contitions and formulas.

The root of a checklist should be a checklist object. In JSON format just {...}. 

<checklist>
      ...
</checklist>

<name> is the only required property on this schema. You can also add a description to give a long explanation of what type of project the checklist applies to. 

<checklist>
    <name>New Residential Structure (Single family)</name>
    <desc>Minimum requirements for submittal of new residential single family structure.</desc>
    ...
</checklist>

Fields section

Add a <fields> section to specify which external values apply to this project type.

<checklist>
    ...
    <fields>
      ...
    </fields>
    ...
</checklist>

Refer to the Fields format for details on the structure of the fields section.

Documents section

Add a <documents> section to specify which documents are required for this permit type.

<checklist>
    ...
    <documents>
      ...
    </documents>
    ...
</checklist>

Document

Documents is an array of <document> objects with the following structure.

Property Description Notes
name Name or short description for the document. Required field. Unique id.
desc Longer description for the document, with instructions or other help text.
sign
  • true denotes that the document needs to be sealed by design professional
  • false document not needs to be sealed by design professional
 Possible values: true, false. Default value: false
conditions List of rules that define when this document is required in the conditions format. Document is required if no conditions specified.
code Short code to use when renaming the document using the applicable naming conventions. 
<documents>
    ...
    <document>
      <name>Building Application</name>
      <code>APP B</code>
    </document>
    <document>
      <name>Wind Load Calculations</name>
      <code>WIND CALCS</code>
      <sign>true</sign>
    </document>
    <document>
      <name>Notice of Commencement</name>
      <code>NOC</code>
    </document>
    <document>
      <name>Truss Drawings</name>
      <code>TRUSS</code>
      <sign>true</sign>
    </document>
    <document>
      <name>Owner Builder Disclosure Statement</name>
      <code>OB</code>
      <conditions>
        <condition>
          <field>OWNERBUILDER</field>
          <value>true</value>
        </condition>
      </conditions>
    </document>
    ...
</documents>

Discipline

Disciplines is an array of <discipline> objects with the following structure.

Property Description Notes
name Name or short description for the document. Required field. Unique id.
desc Longer description for the document, with instructions or other help text.
sign
  • true denotes that the drawing type needs to be sealed by design professional
  • false drawing type not needs to be sealed by design professional
 Possible values: true, false. Default value: false
stamp_box Stamp box object that defines the size and position of the box needed for the jurisdiction to apply their stamp.
conditions List of rules that define when this discipline is required in the conditions format. Discipline is required if no conditions specified.
code Short code to use when renaming the drawing using the applicable naming conventions.

Stamp box

Property Description Notes
size Size of the stamp box in inches. Two numeric value separated by an "x". Ex. 10x5
pos Location in the sheet where the stmp box will be placed. Possible values: top-right, top-left, bottom-right, bottom-left,top-center,bottom-center. 
<disciplines>
    ...
    <discipline>
      <name>Cover Sheet</name>
      <code>CS</code>
      <sign>true</sign>
      <stamp_box>
        <size>10x5</size>
        <pos>top-right</pos>
      </stamp_box>
    </discipline>
    <discipline>
      <name>Architectural</name>
      <code>A</code>
      <sign>true</sign>
    </discipline>
    <discipline>
      <name>Electrical</name>
      <code>E</code>
    </discipline>
    <discipline>
      <name>Structural</name>
      <code>S</code>
      <sign>true</sign>
    </discipline>
    <discipline>
      <name>Demolition</name>
      <code>DE</code>
      <conditions>
        <condition>
          <field>DEMO</field>
          <value>true</value>
        </condition>
      </conditions>
    </discipline>
    ...
</disciplines>

Fee

Fees is an array of <fee> objects with the following structure.

Property Description Notes
name Name or short description for the fee. Required field. Unique id.
desc Longer description for the fee, with instructions or explanation of the nature of the fee.
value The numeric fee amount or a text in the formula format with the formula to be used to calculate the fee. Ex. ((VALUATION - 1000) / 1000) * 5
conditions List of rules that define when this fee applies in the conditions format. Fee is assumed to apply if no conditions specified. 
<fees>
    ...
    <fee>
      <name>Sample Flat Fee</name>
      <value>50</value>
    </fee>
    <fee>
      <name>Sample Formula With Condition Fee</name>
      <value>((VALUATION - 1000) / 1000) * 5</value>
      <conditions>
        <condition>
          <field>VALUATION</field>
          <op>G</op>
          <value>1000</value>
        </condition>
      </conditions>
    </fee>
    ...
</fees>
