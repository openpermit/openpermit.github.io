Schemas

Open data schemas to concisely specify permit requirements.

Overview

Simple schemas for complex permit data representation.

Introduction

This section defines a set of schemas to represent different types of permit related data. Complex relationships and permit process details can be represented using a declarative and structured approach. A format is also defined to represent a series of external inputs or fields that capture specific project information. In many of the higher level schemas, these fields are combined with rules and formulas to estimate things like specific permit requirements and applicable fees. All this information is represented in a declarative fashion and can be stored as data so that a generic software can be used to process the applicable set of schemas and determine the outputs for specific cases by just receiving as input a few values unique to a given permit.

Schema

Currently we have not defined any formal schema specs or namespaces. These may come in the future as the spec matures.

Reusable Schemas

Introduction

This section defines a set of schemas, conventions and data formats that are used in building higher level schemas for a variety of purposes.

Fields is a declarative schema that can be used to specify miscellaneous required information and the type of that information. Fields can be used to gather information from users, to generate user interfaces and to filter requiremens according to formulas and conditions specified. Formulas employ a simple and familiar syntax to represent rules, calculations and transformations much like the syntax of an Excel spreadsheet. Conditions are properties that allow for the definition of complex logic rules. They can be used to specify dependencies or alternative workflows of a given process or set of requirements.

Fields

Think of fields as variables in a programming language or fillable fields in a Adobe PDF document. Fields are included in different higher level formats to represent external information that is needed to be able to accomplish a task, filter a set of requirements or simply evaluate a formula.

Open311

If you are familiar with the Open311 spec, fields serve a similar purpose as the attributes used to define services.

Fields

Fields is an array of {field} objects with the following structure.

Property Description Notes
name Name for the field. This is a string with no spaces, much like a variable name. Required property. Unique id.
label Short description or label for the field.
desc Long description with instructions or other help text
type Denotes the type of this field.
  • boolean a true/false or yes/no value.
  • string a string of characters without line breaks.
  • text a string of characters that may contain line breaks.
  • numeric a numeric value.
  • list a set of predefined values (specified in this field) where only one value may be selected.
  • multilist a set of predefined values (specified in this field) where several values may be selected.
  • datetime the input generated must be able to transform into a valid ISO 8601 date.
 Possible values: boolean, string, text, numeric, list, multilist, dtatime.
items If the <type> property is list or multilist this property is an array of <item> strings that specify the possible values. Required if <type> is present.

Example

[
    {
      name: 'VALUATION',
      type: 'numeric',
      label: 'Project Valuation'
    },
    {
      name: 'OWNERBUILDER',
      type: 'boolean',
      label: 'Is owner builder?'
    },
    {
      name: 'FNAME',
      type: 'string',
      label: 'First Name'
    },
    {
      name: 'PROJDESC',
      type: 'text',
      label: 'Project Description'
    },
    {
      name: 'STYPE',
      type: 'list',
      label: 'Structure Type'
      items: [
         'Residential Single family',
         'Residential Multi-family',
         'Commercial',
         'Mobile Home'
      ]
    },
    ...
]

Formulas

Formulas can be used in PRDF to declaratively specify dynamic expressions that need to be calculated on the fly such as rules, mathematical expressions or string transformations. Each specific PRDF schema defines which properties if any are of type formula, schemas can also define a range of valid values that those formulas can evaluate to. Formulas work pretty much like Excel formulas, but there are restrictions on where these formulas can appear and what values they can evaluate to.

In fact, formulas must follow the same syntax as Excel Formulas part of OOXML Standard (ECMA-376, ISO/IEC 29500-1:2008). If you prefer you can also follow OpenOffice formulas OpenDocument v1.2 OpenFormula (ODF v1.2 OASIS Standard). Only a subset of these specifications is needed, more work to be done exactly on what will be supported by Open Permit.

Formula tips

If you are not familiar with Excel formulas then a Google search should bring plenty of documentation on the subject.

Examples

2 + 3 * 4

((VALUATION - 1000) / 1000) * 5

MAX(SQFEET * 10, 2500)

Conditions

Conditions can be used to declaratively specify rules that dictate when some specific requirement or workflow step applies to a process. Conditions are added to specific items within different schemas to be able to filter them depending on the specific values provided by fields. Conditions are specified using the formula syntax and are attached to an item by adding the <condition> property.

When <condition> is supported on an item, you simply specify a formula that evaluates to a boolean value.

Example

{
    ...
    condition: 'VALUATION > 20000'
    ...
}

Required items

When items that support <condition> do not have one attached, it is assumed by convention that the item is required.

Checklist schema

Structure

The checklist schema can be used to publish specific permit requirement information including documents to be submitted, plan review disciplines that apply if any, applicable subpermits and fees. This format supports the definition of conditions and formulas that are tied to requirements so that the requirements can be filtered according to project details. Fields can also be defined that specify miscelaneous required information needed to filter the requiremens according to the contitions and formulas. Field within a checklist can be grouped into pages and sections so that the information can be grouped for presentation purposes.

The {checklist} is an object with the following fields:

Property Description Notes
name Name or short description for the checklist. Required field.
desc Longer description for the checklist, with instructions or other help text.
documents Array of {document}[] objects.
    document.name Name or short description for the document. Required field. Unique id.
    document.desc Longer description for the document, with instructions or other help text.
    document.sign
  • true denotes that the document needs to be sealed by design professional
  • false document not needs to be sealed by design professional
 Possible values: true, false. Default value: false
    document.condition Logic expression that defines when this document is required. Expression must follow the formula syntax. Document is required if no condition property is specified.
    document.code Short code to use when renaming the document using the applicable naming conventions.
disciplines Array of {discipline}[] objects.
    discipline.name Name or short description for the discipline. Required field. Unique id.
    discipline.desc Longer description for the document, with instructions or other help text.
    discipline.sign
  • true denotes that the drawing type needs to be sealed by design professional
  • false drawing type not needs to be sealed by design professional
 Possible values: true, false. Default value: false
    discipline.stamp_box {Stamp box} object that defines the size and position of the box needed for the jurisdiction to apply their stamp.
      stamp_box.size Size of the stamp box in inches separated by an x. Ex. 10x5
      stamp_box.position Location in the sheet where the stamp box will be placed. Possible values: top-right, top-left, bottom-right, bottom-left, top-center, bottom-center
    discipline.condition Logic expression that defines when this discipline is required. Expression must follow the formula syntax. Discipline is required if no condition property is specified.
    discipline.code Short code to use when renaming the drawing using the applicable naming conventions.
permits Array of {permittype}[] objects.
    permittype.id The unique id for the permit type. Required field.
    permittype.name Name or short description for the permit. Required field.
    permittype.desc Longer description for the permit, with instructions or other help text.
    permittype.condition Logic expression that defines when this permit is required. Expression must follow the formula syntax. Permit is required if no condition property is specified.
fees Array of {fee}[] objects.
    fee.id Unique id or code for this fee. Required field. Unique id.
    fee.name Name or short description for the fee. Required field.
    fee.desc Longer description for the fee, with instructions or explanation of the nature of the fee.
    fee.formula The numeric fee amount or a text in the formula format with the formula to be used to calculate the fee. Ex. ((VALUATION - 1000) / 1000) * 5
    fee.condition Logic expression that defines when this fee is required. Expression must follow the formula syntax. Fee is assumed to apply if no condition property is specified.
pages Array of {page}[] objects. Pages contain fields which specify which information is needed to dynamically filter the checklist.
    page.label Short description for this page. Required field.
    page.sections Array of {section}[] objects. Sections to visually subdivide this page into. Required field.
        section.label Short description for this section. Required field.
        section.fields Refer to the Fields schema for details on the structure of the fields. Required field.

Example

{
    name: 'New Residential Structure (Single family)',
    desc: 'Minimum requirements for a new single family project.',
    documents: [
      {
        name: 'Wind Load Calculations',
        code: 'WIND CALCS',
        sign: true
      },
      {
        name: 'Notice of Commencement',
        code: 'NOC'
      }, 
      {
        name: 'Truss Drawings',
        code: 'TRUSS',
        sign: true
      }, 
      {
        name: 'Owner Builder Disclousure Statement',
        code: 'TRUSS',
        condition: 'OWNERBUILDER = true'
      }, 
    ],

    ...

    disciplines: [
      {
        name: 'Cover Sheet',
        code: 'CS',
        sign: true,
        stamp_box: {
          size: '10x5',
          position: 'top-right'
        }
      },
      {
        name: 'Architectural',
        code: 'A',
        sign: true
      }, 
      {
        name: 'Structural',
        code: 'S',
        sign: true
      },
      {
        name: 'Electrical',
        code: 'E',
        sign: true
      },  
      {
        name: 'Demolition',
        code: 'DE',
        sign: true,
        condition: 'DEMO = true'
      },
    ],

    ...

    permits: [
      {
        name: 'Residential Electrical',
        id: '0001-Building-Residential-electrical-NA',
        desc: 'Permit for residential electrical work.'
      },
      {
        name: 'Residential Structural',
        id: '0001-Building-Residential-Structural-NA',
        desc: 'Permit for residential structural work.'
      }, 
      {
        name: 'Residential Demolition',
        id: '0001-Buildinhg-Residential-Demolition-NA',
        desc: 'Permit for residential demolition work.',
        condition: 'DEMO = true'
      },  
    ],

    ...

    fees: [
      {
        id: 'BLD_01',
        name: 'Application Fee',
        formula: '25',
      },
      {
        id: 'BLD_02',
        name: 'Permit Fee (Valuation)',
        formula: '(VALUATION - 1000) / 1000 * 25',
      }, 
      {
        id: 'BLD_03',
        name: 'Demolition Fee',
        formula: '100',
        condition: 'DEMO = true'
      },  
    ],

    ...

    pages: [
      {
        label: 'Project Info',
        sections: [
          {
            label: 'Details',
            fields: [
              {
                name: 'VALUATION',
                type: 'numeric',
                label: 'Project Valuation'
              },
              {
                name: 'OWNERBUILDER',
                type: 'boolean',
                label: 'Is owner builder?'
              }, 
              {
                name: 'DEMO',
                type: 'boolean',
                label: 'Demolition needed?'
              },   
            ]
          }
        ],
      }
    ]
}
